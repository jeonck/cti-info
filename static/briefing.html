01 — 배경 · 목적

왜 만들었나

사이버 위협 정보를 온톨로지·지식그래프로 구조화해 잠재적 공격을 분석하고 상호운용성(STIX 등 표준)을 확보하는 것이 최종 목표다. 그 재료가 되는 CVE·TTP·위협행위자 동향은 매일 수십 건씩 흩어져 쏟아지는데, 사람이 매일 훑어 우선순위를 매기는 일은 지속하기 어렵다.

이 파이프라인이 대신하는 일

  • 수집 — 12개 RSS·3개 Reddit·4개 HN 쿼리·3개 GitHub 검색을 매일 자동으로 훑는다.
  • 판정 — Claude가 context.md에 적힌 연구 관심사를 기준으로 각 항목의 행동 가치를 매긴다.
  • 필터링 — "무관" 판정은 포스트를 만들지 않는다. 남는 것은 실제로 볼 가치가 있는 항목뿐.
  • 기록 — 판정마다 근거와 다음 액션(예: ATT&CK 매핑, STIX 객체화)을 함께 남긴다.

왜 정적 사이트인가

별도 서버·DB 없이 GitHub Actions가 크론 실행 → Hugo 빌드 → GitHub Pages 배포까지 전부 처리한다. 운영 비용은 사실상 0원이고, 판정 결과는 verdict·status·tag 축으로 바로 탐색할 수 있다.

03 — 판정 컨텍스트

context.md — 행동 판정의 유일한 기준

여기 없는 주제는 원칙적으로 "무관" 또는 "학습"으로 판정한다. 현재 SOC처럼 지켜야 할 인프라는 없으므로, "즉시조치"의 기준은 구조화·분석 관점에서 즉시 확인해야 할 자료로 재정의했다.

🔥 즉시조치

활발히 익스플로잇 중인 CVE, 긴급 패치가 필요한 자료

예 — CISA KEV 신규 등재, PoC 공개된 Critical CVE

📌 백로그

추적할 가치는 있으나 긴급하지 않은 항목

예 — 신규 CVE 공개(익스플로잇 미확인), 벤더 보안 권고

📚 학습

구조화·TTP·표준·위협행위자 동향 관심 분야

예 — ATT&CK 매핑 사례, STIX/TAXII·온톨로지 논의

⋯ 무관

그 외 전부 — 포스트를 생성하지 않는다

예 — 제품 마케팅, CTI와 무관한 일반 IT 뉴스

그 외

05 — 판정 결과 예시

실제 dry-run 판정 1건

The Hacker News의 "휴면 GitHub 계정을 이용한 정찰 캠페인" 기사가 어떻게 판정·기록되는지.

📚 학습 status: 대기 source: The Hacker News
유령 GitHub 계정을 이용한 기업 조직 정찰 캠페인
#threat-actor-ttp #reconnaissance #github-api-abuse
근거
GitHub API를 통한 기업 조직 정찰 캠페인은 위협 행위자 TTP(정찰 단계) 분석 및 ATT&CK 매핑 연구에 직접 해당
액션
보고서 원문 확인 후 T1589 / T1590(Gather Victim Identity / Network Information) 하위 기법으로 STIX Course of Action 객체 초안 작성

매일 자동 수집·판정

일시중지 · 재개

수동 워크플로(Pipeline Control) 또는 스크립트로 수집만 중지 가능, 배포 파이프라인은 별도로 유지

Weekly

금요일 리뷰

처리한 백로그·학습 항목을 완료 처리하고, 판정이 어긋나면 context.md를 보강

알려진 제약

  • Reddit API가 GitHub Actions IP에서 종종 403 — 소스별 오류 격리로 다른 소스는 정상 수집
  • GitHub 검색은 created:>N일 조건상 0건인 날이 흔함 (정상 동작)
  • 크레딧 부족·인증 만료 시 즉시 중단하고 워크플로를 실패로 표시 (의도된 동작)

07 — 범위와 다음 단계

지금은 경량, 확장은 열어둔다

초기 구상은 OpenCTI/MISP 연동, STIX/TAXII 정규화, VirusTotal·GreyNoise 등 enrichment API를 포함한 본격 CTI 플랫폼이었다. 현재 파이프라인은 그 전 단계 — 텍스트 판정 기반의 경량 트리아지 레이어로 의도적으로 범위를 좁혔다.

현재 범위

  • verdict/status/tag 축 탐색
범위 밖 (확장 후보)
  • OpenCTI / MISP 연동, 그래프 저장소
